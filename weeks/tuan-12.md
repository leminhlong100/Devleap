**Java Core must-knows**

**Spring essentials**

DI/IoC là trái tim của Spring — container quản lý bean và inject dependency. @RestController + @RequestMapping xây REST API. Spring Data JPA tạo repository tự động từ interface. @Transactional quản lý transaction theo declarative. Hiểu rõ vấn đề N+1 và cách xử lý (fetch join, @EntityGraph) là điểm hỏi rất hay gặp.

**Spring AI + modern stack**

Đây là điểm khác biệt (differentiator) của bạn so với ứng viên khác. ChatClient để gọi LLM, RAG (Retrieval-Augmented Generation) kết hợp vector store cho câu trả lời theo ngữ cảnh, và streaming response. Biết tích hợp AI vào backend Spring Boot là kỹ năng hiện đại, đáng giá và gây ấn tượng mạnh trong phỏng vấn.

**Common interview traps**

Những bẫy hay gặp cần thuộc lòng: HashMap internals (bucket, hash, treeify khi quá 8), == so sánh reference vs equals so sánh nội dung, String immutability và String pool, phân biệt final/finally/finalize, checked vs unchecked exception, và fail-fast iterator (ném ConcurrentModificationException) vs fail-safe. Trả lời chắc nhóm này thể hiện chiều sâu.

```java
import java.util.*;
import java.util.stream.*;

// equals/hashCode contract: bằng nhau theo equals => cùng hashCode
record Employee(Long id, String name, double salary) {
    // record tự sinh equals/hashCode/toString chuẩn theo mọi field
}

class RecallDemo {
    public static void main(String[] args) {
        List<Employee> emps = List.of(
            new Employee(1L, "An", 1200),
            new Employee(2L, "Binh", 1800),
            new Employee(3L, "Chi", 1500)
        );

        // Stream: filter + map + collect
        List<String> highPaid = emps.stream()
            .filter(e -> e.salary() >= 1500)
            .map(Employee::name)
            .collect(Collectors.toList());
        System.out.println(highPaid); // [Binh, Chi]

        // Optional: tránh NullPointerException
        Optional<Employee> top = emps.stream()
            .max(Comparator.comparingDouble(Employee::salary));
        top.ifPresent(e -> System.out.println("Top: " + e.name()));

        // equals/hashCode hoạt động đúng trong Set
        Set<Employee> set = new HashSet<>(emps);
        System.out.println("Size: " + set.size()); // 3
    }
}
```

### 🎤 Câu Hỏi Phỏng Vấn — Dễ → Khó (7 câu)

**Dễ · Phân biệt == và equals() trong Java?**

== so sánh reference (hai biến có trỏ tới cùng một object trong bộ nhớ không) với object, và so sánh giá trị với primitive. equals() so sánh nội dung logic và có thể override để định nghĩa "bằng nhau" theo ý nghĩa nghiệp vụ. Ví dụ hai String "abc" tạo bằng new có thể == trả về false nhưng equals() trả về true vì cùng nội dung. Khi override equals() phải override luôn hashCode() để giữ contract.

**Dễ · Vì sao String trong Java là immutable?**

String immutable vì nhiều lý do: cho phép String Pool tái sử dụng object (tiết kiệm bộ nhớ), an toàn cho cache hashCode (dùng nhiều trong HashMap key), thread-safe tự nhiên (không thể bị thay đổi giữa các thread), và an toàn bảo mật (tham số như đường dẫn file, kết nối mạng không bị sửa sau khi kiểm tra). Mọi thao tác "thay đổi" String thực ra tạo ra object String mới.

**Dễ · Phân biệt final, finally và finalize?**

Ba khái niệm không liên quan ngoài cái tên: **final** là modifier — biến final không đổi giá trị, method final không override được, class final không kế thừa được. **finally** là block trong try-catch luôn chạy dù có exception hay không, dùng để dọn tài nguyên. **finalize()** là method được GC gọi trước khi thu hồi object (đã deprecated từ Java 9, không nên dùng — thay bằng try-with-resources/Cleaner).

**Trung · Phân biệt fail-fast và fail-safe iterator?**

Fail-fast iterator (ArrayList, HashMap) ném ConcurrentModificationException ngay khi phát hiện collection bị sửa cấu trúc trong lúc duyệt (qua biến modCount). Fail-safe iterator (CopyOnWriteArrayList, ConcurrentHashMap) duyệt trên một bản sao/snapshot nên không ném exception khi collection bị sửa, nhưng có thể không thấy thay đổi mới nhất và tốn bộ nhớ hơn. Để sửa an toàn khi duyệt fail-fast, dùng Iterator.remove().

**Trung · HashMap hoạt động bên trong như thế nào?**

HashMap dùng mảng các bucket. Khi put, nó tính hashCode() của key rồi áp một hàm trộn (spread) để xác định index bucket. Nếu nhiều key rơi vào cùng bucket (collision), chúng được nối thành linked list; từ Java 8, khi một bucket vượt 8 phần tử và bảng đủ lớn, list được chuyển thành cây đỏ-đen (treeify) để tra cứu O(log n). Khi số phần tử vượt load factor (mặc định 0.75), bảng được resize gấp đôi và rehash. get tính hash, tìm bucket, dùng equals() để khớp key chính xác.

**Khó · @Transactional có những kiểu propagation nào và ý nghĩa?**

Propagation quy định cách transaction hiện tại tương tác với transaction đang tồn tại. Các kiểu chính: **REQUIRED** (mặc định) — tham gia transaction hiện có, nếu chưa có thì tạo mới. **REQUIRES_NEW** — luôn tạo transaction mới, tạm dừng cái cũ (dùng cho thao tác độc lập như ghi log/audit). **SUPPORTS** — tham gia nếu có, không thì chạy non-transactional. **MANDATORY** — bắt buộc phải có sẵn transaction, không thì ném exception. **NESTED** — tạo savepoint lồng trong transaction cha. Lưu ý self-invocation (gọi method @Transactional cùng class) không kích hoạt proxy nên transaction không áp dụng.

**Khó · Mô tả vòng đời (lifecycle) của một Spring bean?**

Vòng đời bean: (1) Container instantiate bean qua constructor. (2) Populate properties / inject dependencies. (3) Gọi các Aware interface (BeanNameAware, ApplicationContextAware). (4) BeanPostProcessor.postProcessBeforeInitialization. (5) @PostConstruct hoặc afterPropertiesSet() (InitializingBean) hoặc init-method. (6) BeanPostProcessor.postProcessAfterInitialization (đây là nơi AOP proxy được bọc). (7) Bean sẵn sàng sử dụng. (8) Khi container shutdown: @PreDestroy hoặc destroy() (DisposableBean) hoặc destroy-method. Bean singleton sống suốt đời container; prototype không được container quản lý phần hủy.

1. Khi override `equals()` trong Java, vì sao bắt buộc override luôn `hashCode()`?

2. Từ Java 8, khi một bucket trong HashMap vượt 8 phần tử và bảng đủ lớn, điều gì xảy ra?

   - [ ] HashMap ném ConcurrentModificationException
   - [x] Linked list trong bucket được chuyển thành cây đỏ-đen (treeify) để tra cứu O(log n)

   💡 Treeify biến danh sách liên kết thành red-black tree, cải thiện tra cứu từ O(n) xuống O(log n) khi collision nhiều.

### 📖 Lý Thuyết Cốt Lõi

**Microservices must-knows**

So với monolith: microservices cho phép scale và deploy độc lập từng service nhưng đổi lại độ phức tạp vận hành. Các thành phần cốt lõi: API Gateway (single entry point), circuit breaker (chống cascade failure), service discovery (Eureka/Consul). Thách thức phân tán: distributed transaction (Saga), eventual consistency, network latency, và debugging khó hơn (cần distributed tracing).

**JVM + GC**

Memory areas: Heap (Young: Eden+Survivor, Old gen), Metaspace, Stack (mỗi thread), PC register. GC algorithms: Serial, Parallel, G1 (mặc định, cân bằng throughput/pause), ZGC/Shenandoah (low-latency, pause < 1ms). Memory leak xảy ra khi giữ reference không cần (static collection phình to, listener không gỡ). Đây là chủ đề performance interview hay hỏi.

**Design Patterns**

Cần nhận diện và áp dụng được: Singleton (một instance duy nhất, lưu ý thread-safety), Factory (tạo object không lộ class cụ thể), Builder (xây object phức tạp từng bước), Strategy (đổi thuật toán runtime), Observer (publish-subscribe, một thay đổi báo nhiều listener). Quan trọng là biết khi nào dùng, không lạm dụng pattern.

**System Design framework**

Quy trình chuẩn: (1) Làm rõ requirements (functional + non-functional). (2) Ước lượng scale (QPS, storage, bandwidth). (3) High-level design (vẽ component chính). (4) Deep-dive vào phần quan trọng. (5) Xác định bottleneck và cách giải. Công cụ thường dùng: caching (Redis), horizontal scaling, load balancer, chọn DB (SQL vs NoSQL), message queue cho async.

### 🎤 Câu Hỏi Phỏng Vấn — Dễ → Khó (7 câu)

**Dễ · Microservices và monolith khác nhau, đánh đổi gì?**

Monolith là một ứng dụng đơn khối: dễ phát triển ban đầu, deploy đơn giản, debug dễ vì gọi trong process, nhưng khó scale từng phần và khi lớn thì cồng kềnh, đổi một chỗ phải deploy cả hệ thống. Microservices chia thành service nhỏ độc lập: scale và deploy riêng từng service, đội nhóm sở hữu riêng, công nghệ linh hoạt; nhưng đánh đổi bằng độ phức tạp vận hành (network, distributed transaction, monitoring, deployment). Chọn microservices khi quy mô lớn, nhiều team, và tổ chức đủ năng lực vận hành.

**Dễ · Circuit breaker có những trạng thái nào?**

Circuit breaker có 3 trạng thái: **CLOSED** — bình thường, request đi qua; đếm tỉ lệ lỗi. **OPEN** — khi lỗi vượt ngưỡng, breaker mở và chặn request ngay lập tức (fail fast, trả fallback) để tránh cascade failure và cho service lỗi thời gian hồi phục. **HALF-OPEN** — sau một khoảng chờ, cho qua vài request thử; nếu thành công thì về CLOSED, nếu thất bại thì về OPEN. Resilience4j hiện thực mô hình này trong Spring.

**Dễ · JVM có những vùng bộ nhớ nào?**

Các vùng chính: **Heap** — chứa object, chia Young Generation (Eden + 2 Survivor, nơi object mới sinh) và Old Generation (object sống lâu). **Metaspace** — chứa metadata class (thay PermGen từ Java 8, nằm ngoài heap). **Stack** — mỗi thread một stack chứa frame của method (biến local, tham số). **PC Register** — con trỏ lệnh mỗi thread. **Native Method Stack** — cho method native. Heap dùng chung mọi thread; Stack và PC register riêng từng thread.

**Trung · So sánh G1 và ZGC garbage collector?**

G1 (Garbage First, mặc định từ Java 9) chia heap thành nhiều region, ưu tiên thu region nhiều rác trước, cân bằng giữa throughput và pause time, phù hợp đa số ứng dụng với heap vừa-lớn; pause cỡ vài chục đến trăm ms. ZGC là collector low-latency, thiết kế cho pause cực ngắn (< 1ms thậm chí với heap hàng terabyte), làm phần lớn việc concurrent với app thread, phù hợp ứng dụng nhạy độ trễ (trading, real-time). Đổi lại ZGC tốn CPU/bộ nhớ overhead hơn. Chọn G1 cho cân bằng chung, ZGC khi cần pause cực thấp.

**Trung · Làm sao để Singleton thread-safe?**

Có vài cách: (1) **Eager initialization** — tạo instance ngay khi load class (static final), thread-safe tự nhiên nhưng tạo dù không dùng. (2) **Double-checked locking** — biến volatile + kiểm tra hai lần trong synchronized block, tránh lock mỗi lần get. (3) **Bill Pugh / static holder** — dùng inner static class, JVM đảm bảo lazy + thread-safe khi load holder, cách được khuyên dùng nhất. (4) **Enum singleton** — đơn giản nhất, an toàn cả với serialization và reflection. Trong Spring thì bean singleton mặc định do container quản lý.

**Khó · Phân biệt Strategy pattern và Factory pattern?**

Khác nhau ở mục đích. **Factory** là creational pattern — lo việc tạo object: client xin một object qua factory mà không cần biết class cụ thể, factory quyết định trả về implementation nào. **Strategy** là behavioral pattern — lo việc chọn hành vi: định nghĩa một họ thuật toán có thể hoán đổi runtime, client giữ reference tới một strategy và gọi nó, có thể thay strategy khác lúc chạy. Thường dùng kết hợp: factory tạo ra strategy phù hợp rồi inject vào context. Tóm lại Factory = tạo cái gì, Strategy = làm thế nào.

**Khó · Cách tiếp cận một bài system design trong phỏng vấn?**

Theo framework có cấu trúc: (1) **Clarify requirements** — hỏi để xác định functional (tính năng) và non-functional (scale, latency, availability, consistency), đừng vội vẽ. (2) **Estimate scale** — DAU, QPS read/write, storage, bandwidth để định cỡ. (3) **High-level design** — vẽ API, các component chính (LB, app, cache, DB, queue) và luồng dữ liệu. (4) **Deep-dive** — đào sâu phần quan trọng/khó nhất (data model, sharding, cache strategy). (5) **Identify bottlenecks & tradeoffs** — chỉ ra điểm nghẽn và cách giải (caching, replication, sharding), nêu rõ đánh đổi. Quan trọng là tư duy ra tiếng và giao tiếp liên tục với interviewer.

### 📖 Lý Thuyết Cốt Lõi

**STAR method**

Cấu trúc trả lời behavioral: **S**ituation (bối cảnh ngắn gọn), **T**ask (nhiệm vụ/trách nhiệm của bạn), **A**ction (hành động cụ thể BẠN làm — dùng "I" không "we"), **R**esult (kết quả, quantify bằng con số nếu được + bài học). **Common behavioral questions**

Chuẩn bị trước cho: "Tell me about yourself", thử thách lớn nhất, một lần thất bại, xung đột với đồng nghiệp, vì sao chọn công ty này, điểm mạnh/điểm yếu, một lần bạn dẫn dắt/chủ động. Hầu hết câu hỏi behavioral đều xoay quanh nhóm này — chuẩn bị sẵn câu chuyện STAR cho từng nhóm là chìa khóa.

**Storytelling for impact**

Dùng ví dụ cụ thể, có thật thay vì nói chung chung. Làm rõ vai trò và đóng góp riêng của bạn trong nhóm. Luôn nêu lessons learned và growth mindset — interviewer đánh giá cao người biết rút kinh nghiệm và phát triển. Giữ câu chuyện ngắn gọn (1.5-2 phút), tập trung vào Action và Result.

**Questions to ask interviewer**

Cuối buổi luôn có "Do you have any questions?" — đừng nói "No". Chuẩn bị 4-5 câu thông minh: cấu trúc team, tech stack và quy trình làng việc, cơ hội phát triển/học hỏi, thách thức lớn nhất của team hiện tại, định nghĩa thành công ở vị trí này. // CODING INTERVIEW — THINK-ALOUD COMMENTARY
```java
// Bài: Two Sum — trả về index hai số cộng lại bằng target.

// STEP 1 — CLARIFY (nói ra cho interviewer):
//   "Mảng có sắp xếp không? Có số âm không? Có đúng một
//    nghiệm không? Có dùng lại cùng phần tử không?"
//   Giả định: chính xác 1 nghiệm, không dùng lại 1 index.

// STEP 2 — APPROACH:
//   "Brute-force là 2 vòng lặp O(n^2). Tốt hơn: dùng HashMap
//    lưu (value -> index). Với mỗi số x, tìm target - x đã
//    thấy chưa. Đổi time O(n) lấy space O(n)."

// STEP 3 — COMPLEXITY: Time O(n), Space O(n).

import java.util.*;

class Solution {
    public int[] twoSum(int[] nums, int target) {
        Map<Integer, Integer> seen = new HashMap<>();
        for (int i = 0; i < nums.length; i++) {
            int need = target - nums[i];
            // "Nếu phần bù đã xuất hiện trước đó -> tìm được cặp"
            if (seen.containsKey(need)) {
                return new int[]{ seen.get(need), i };
            }
            seen.put(nums[i], i);
        }
        return new int[]{}; // theo đề sẽ không tới đây
    }
}

// STEP 4 — TEST (nói ra):
//   nums=[2,7,11,15], target=9 -> i=0 put(2,0); i=1 need=2
//   đã có -> trả [0,1]. Edge: mảng rỗng -> trả []. OK.
```

```java
// CAPSTONE — signature piece kết tinh cả 12 tuần:
// Spring Boot + JPA + Redis + Spring AI + Security, gắn vào nhau.
// 🎉 Đây là minh chứng bạn đã đi được bao xa. Chúc mừng!

@RestController
@RequestMapping("/api/assistant")
@RequiredArgsConstructor
public class AssistantController {

    private final AssistantService service;

    // Security: chỉ user đã xác thực JWT mới gọi được (W4-W5)
    @PreAuthorize("isAuthenticated()")
    @PostMapping("/ask")
    public ResponseEntity<AnswerDto> ask(@Valid @RequestBody AskRequest req,
                                          @AuthenticationPrincipal UserDetails user) {
        AnswerDto answer = service.answer(user.getUsername(), req.question());
        return ResponseEntity.ok(answer);
    }
}

@Service
@RequiredArgsConstructor
public class AssistantService {

    private final ChatClient chatClient;          // Spring AI (W6)
    private final QueryLogRepository logRepo;      // Spring Data JPA (W3)
    private final StringRedisTemplate redis;       // Redis caching (W7)

    @Transactional
    public AnswerDto answer(String username, String question) {
        String cacheKey = "qa:" + question.hashCode();

        // 1) Cache-aside: trả nhanh nếu đã hỏi (W7)
        String cached = redis.opsForValue().get(cacheKey);
        if (cached != null) {
            return new AnswerDto(cached, true);
        }

        // 2) Gọi LLM qua Spring AI (W6)
        String reply = chatClient.prompt()
                .user(question)
                .call()
                .content();

        // 3) Lưu cache (TTL 10 phút) + log lịch sử (W3)
        redis.opsForValue().set(cacheKey, reply, Duration.ofMinutes(10));
        logRepo.save(new QueryLog(username, question, Instant.now()));

        return new AnswerDto(reply, false);
    }
}
```

### 🎤 Câu Hỏi Phỏng Vấn — Dễ → Khó (10 câu)

**Dễ · Bốn trụ cột của OOP là gì?**

Bốn trụ cột: **Encapsulation** (đóng gói — giấu trạng thái nội bộ, truy cập qua method), **Inheritance** (kế thừa — class con dùng lại class cha), **Polymorphism** (đa hình — cùng interface, hành vi khác nhau; overriding/overloading), **Abstraction** (trừu tượng — ẩn chi tiết, lộ ra cái cần thiết qua interface/abstract class). Đây là nền tảng tư duy thiết kế hướng đối tượng.

**Dễ · Khi nào dùng ArrayList và khi nào dùng LinkedList?**

ArrayList dựa trên mảng động: truy cập theo index O(1), nhưng chèn/xóa giữa O(n) vì phải dịch phần tử. LinkedList là danh sách liên kết đôi: chèn/xóa ở hai đầu O(1), nhưng truy cập theo index O(n). Thực tế ArrayList tốt cho phần lớn trường hợp (cache-friendly, ít overhead); chỉ dùng LinkedList khi thao tác chèn/xóa nhiều ở đầu/cuối hoặc cần Deque. Mặc định nên chọn ArrayList.

**Trung · @SpringBootApplication làm những gì?**

Nó là meta-annotation gộp 3 cái: **@SpringBootConfiguration** (đánh dấu class cấu hình), **@EnableAutoConfiguration** (tự cấu hình bean dựa trên dependency có trên classpath — trái tim của Spring Boot), và **@ComponentScan** (quét và đăng ký bean trong package hiện tại và con). Nhờ vậy Boot giảm boilerplate cấu hình so với Spring thuần, theo triết lý convention over configuration.

**Trung · Giải quyết vấn đề N+1 query trong JPA như thế nào?**

N+1 xảy ra khi load N entity cha rồi mỗi cha lại trigger thêm 1 query lấy con (do lazy loading), thành N+1 query. Cách giải: dùng **JOIN FETCH** trong JPQL để lấy cha + con trong một query; hoặc **@EntityGraph** khai báo eager cho query cụ thể; hoặc **batch fetching** (@BatchSize) gom nhiều query con thành ít query hơn. Tránh đặt EAGER toàn cục. Luôn bật show-sql để phát hiện N+1 sớm.

**Khó · Thiết kế hệ thống rút gọn URL (URL shortener) ở mức cao?**

(1) Requirements: shorten URL → short code, redirect short → long; non-functional: read >> write, low latency, high availability. (2) Scale: ước lượng QPS (read nhiều hơn write 100:1), storage cho mappings. (3) Design: API POST /shorten + GET /{code}; sinh code bằng base62 encode của ID tự tăng (hoặc hash + chống trùng); lưu mapping trong DB (key-value hợp với NoSQL hoặc SQL có index). (4) Tối ưu: cache hot URL trong Redis để redirect cực nhanh; CDN; read replica. (5) Tradeoff: counter-based đơn giản nhưng lộ thứ tự; hash-based cần xử lý collision. Đây là bài system design kinh điển.

**Khó · Làm sao đảm bảo tính nhất quán dữ liệu giữa các microservice?**

Trong hệ phân tán không thể dùng transaction ACID xuyên service. Giải pháp: **Saga pattern** — chuỗi transaction local, mỗi bước phát event; nếu một bước lỗi thì chạy compensating transaction để hoàn tác (choreography qua event, hoặc orchestration qua một điều phối trung tâm). Chấp nhận **eventual consistency** thay vì nhất quán tức thì. Dùng **Outbox pattern** để đảm bảo lưu DB và publish event nguyên tử. Idempotency cho consumer để xử lý message trùng. Đây là tradeoff cốt lõi khi chuyển từ monolith sang microservices.

- **Q: What is the difference between == and equals(), and why must equals() and hashCode() be overridden together?**  
  A: == compares references (or values for primitives), while equals() compares logical content and can be overridden. If you override equals() but not hashCode(), objects that are "equal" may land in different buckets, breaking HashMap and HashSet — equal objects must return the same hashCode.
- **Q: What is the N+1 query problem in JPA and how do you solve it?**  
  A: It happens when loading N parent entities triggers N additional queries to fetch their lazy children, totaling N+1 queries. Fix it with a JOIN FETCH in JPQL, an @EntityGraph for eager loading on a specific query, or batch fetching with @BatchSize — and avoid global EAGER mappings.
- **Q: How would you design a scalable URL shortener?**  
  A: Clarify requirements (read-heavy, low latency), estimate scale, then expose POST /shorten and GET /{code}. Generate short codes via base62 encoding of an ID (or a collision-checked hash), store mappings in an indexed DB, cache hot URLs in Redis for fast redirects, and add read replicas plus a CDN to scale.
